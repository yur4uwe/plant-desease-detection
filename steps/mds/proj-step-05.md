# Проектне завдання 5: Вибір критеріїв та побудова системи оцінювання якості даних

      Мета виконання завдання: Сформувати систему оцінювання якості даних для обраної бізнес-задачі, визначити релевантні критерії якості, розробити набір метрик та правил перевірки даних, а також виконати початкове оцінювання якості зібраного датасету. Обґрунтувати, як якість даних впливає на якість аналітики, надійність моделі машинного навчання та бізнес-цінність проекту.

План виконання:
1. Визначення ролі якості даних у проекті Необхідно коротко описати:
          - яку бізнес-задачу розв’язує проект;
          - які саме дані використовуються;
          - чому якість даних є критичною для подальших етапів: побудови датасету, EDA, очищення, моделювання, автоматизованого конвеєра.
У цьому підпункті студент має пояснити зв’язок: якість даних → якість ознак → якість моделі → якість бізнес-рішень.

2. Вибір критеріїв якості даних Потрібно вибрати та описати критерії, які є доречними саме для вашого проекту. Рекомендується використати щонайменше 6–8 критеріїв із наведеного переліку:
   - Повнота (Completeness) — чи всі потрібні значення присутні.
   - Коректність / достовірність (Accuracy / Validity) — чи відповідають значення реальності та допустимим форматам.
   - Узгодженість (Consistency) — чи немає суперечностей між полями, таблицями.
   - Унікальність (Uniqueness) — чи немає дублювання сутностей або записів.
   - Актуальність / своєчасність (Timeliness) — чи не є дані застарілими.
   - Цілісність (Integrity) — чи збережені логічні та ключові зв’язки між даними.
   - Релевантність (Relevance) — чи справді атрибути корисні для вирішення задачі.
   - Збалансованість (Balance) — особливо для задач класифікації.
   - Однорідність форматів (Standardization) — єдині одиниці вимірювання, формати дат, категорій, валют, назв.
   - Шумність / наявність аномалій (Noise / Outliers) — чи є підозрілі значення, які потребують аналізу.
Для кожного вибраного критерію треба подати: назву критерію, визначення, чому він важливий саме для цього проекту, як його можна виміряти.

3. Побудова таблиці критеріїв, метрик і правил перевірки Потрібно створити основну таблицю оцінювання якості даних. Рекомендована структура таблиці:
 № Критерій якості Що перевіряється Метрика / спосіб Порогове значення Дії у разі оцінювання порушення

 1 Повнота Частка пропущених % missing values не більше 5% заповнення, значень у критичних видалення, полях повторний збір

 2 Унікальність Дублікати записів % duplicate rows 0–2% дедуплікація

 3 Коректність Формат email / дати / % валідних записів не менше 95% виправлення чисел формату

 4 Узгодженість Відповідність між кількість конфліктів 0 або мінімум логічна перевірка полями

 5 Актуальність Часова релевантність % записів у не менше 90% фільтрація старих записів допустимому даних часовому вікні

4. Визначення критичних та некритичних атрибутів даних Розділити атрибути на:
     - критичні — без них запис непридатний або суттєво втрачає цінність;
     - важливі — бажані для моделі або аналітики;
     - додаткові — допоміжні.
Рекомендовано подати таблицю:

Це допоможе обґрунтувати, чому для різних полів допустимі різні пороги якості.

5. Формування системи бальної оцінки якості даних Необхідно побудувати власну систему оцінювання якості даних, наприклад, у вигляді інтегрального показника. Один із рекомендованих підходів, Для кожного критерію:
            - задається вага критерію 𝑤𝑖;
            - обчислюється оцінка критерію 𝑞𝑖у межах від 0 до 1;
            - інтегральний показник якості: 𝑛 𝑛 𝑄 = ∑ 𝑤𝑖⋅𝑞𝑖, де ∑ 𝑤𝑖 = 1. 𝑖=1 𝑖=1

  Приклад інтерпретації:
   - 0.90–1.00 — висока якість даних;
   - 0.75–0.89 — прийнятна якість;
   - 0.60–0.74 — середня якість, потрібне очищення;
   - < 0.60 — низька якість, дані потребують суттєвого доопрацювання.
Потрібно:
   - або запропонувати власні ваги критеріїв,
   - або обгрунтувати, чому всі критерії рівнозначні.

6. Проведення первинного оцінювання набору даних На основі зібраного датасету треба виконати базову перевірку якості. Мінімально можна оцінити:
   - кількість рядків і стовпців;
   - кількість пропусків;
   - кількість дублікатів;
   - некоректні типи даних;
   - підозрілі або екстремальні значення;
   - частку валідних записів у ключових полях.
Результати слід подати у вигляді: таблиці, короткого текстового аналізу, висновку про поточний стан даних.

7. Аналіз ризиків, пов’язаних із низькою якістю даних Потрібно описати, які проблеми можуть виникнути через неякісні дані, наприклад:
   - зміщення моделі;
   - зниження точності прогнозів;
   - помилкові бізнес-рішення;
   - нестабільність роботи конвеєра;
   - викривлення аналітичних висновків.
Також потрібно запропонувати способи мінімізації: очищення, нормалізація, дедуплікація, валідаційні правила, автоматичні data quality checks у pipeline, повторний збір або заміна джерел даних.

Очікувані результати та критерії оцінювання:

   - (1 бал) Обґрунтування важливості якості даних - описано, чому якість даних критична для проекту, показано її вплив на аналітику, ML-модель і бізнес-рішення.
   - (1 бал) Вибір критеріїв якості даних - обрано релевантні критерії якості, надано їх визначення та пояснено доцільність використання саме в межах даного проекту.
   - (1 бал) Таблиця критеріїв, метрик і порогів - сформовано повну таблицю з критеріями, правилами перевірки, метриками, порогами та діями у разі порушення.
   - (1 бал) Побудова системи інтегрального оцінювання - розроблено власну або адаптовану систему бальної / вагової оцінки якості даних із чіткою логікою обчислення.
   - (1 бал) Первинне оцінювання якості датасету - виконано перевірку набору даних, наведено результати оцінювання та короткий аналіз виявлених проблем.
   - (1 бал) Аналіз ризиків і висновки - описано ризики низької якості даних, запропоновано методи їх мінімізації та зроблено змістовий висновок щодо готовності даних до подальших етапів.
Разом: 6 балів
