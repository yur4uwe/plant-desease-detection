# Проектне завдання 13: Побудова конвеєру даних з автоматичною оцінкою метрик

      Мета виконання завдання: Розробити автоматизований конвеєр даних для задачі машинного навчання, який забезпечує послідовне опрацювання вхідних даних, навчання або застосування моделі, обчислення ключових метрик якості, збереження результатів оцінювання та формування висновків щодо придатності моделі для бізнес-задачі. Конвеєр повинен бути придатним для повторного запуску, масштабування та подальшої інтеграції в повний ML/ETL-процес.

План виконання:
1. Визначення призначення конвеєру Необхідно описати:
          - яку саме бізнес-задачу вирішує конвеєр;
          - які дані надходять на вхід;
          - який результат формується на виході;
          - на якому етапі конвеєр виконує оцінювання якості. Студент має чітко вказати тип задачі: класифікація; регресія; кластеризація; ранжування; виявлення аномалій; рекомендаційна задача.​

2. Проектування структури конвеєру Потрібно побудувати логічну або блок-схему конвеєру, яка містить такі модулі:
          - завантаження даних;
          - первинна валідація;
          - очищення та підготовка;
          - трансформація ознак;
          - розділення на train / validation / test;
          - навчання або застосування моделі;
          - автоматичне обчислення метрик;
          - збереження результатів;
          - формування звіту. Бажано подати конвеєр у вигляді:
          - структурної схеми;
          - UML activity diagram;
          - data flow diagram;
          - або схеми модулів у Jupyter Notebook / Python-проекті.​

3. Реалізація автоматичного запуску етапів Необхідно реалізувати код, який:
          - виконує всі етапи послідовно;
          - не потребує ручного втручання між кроками;
          - може бути повторно завантажений на нових даних;
          - має зрозумілу структуру модулів або функцій. Рекомендовано:
          - оформити pipeline через Python-функції або класи;
          - використовувати sklearn.pipeline.Pipeline, ColumnTransformer, власні трансформери;
          - передбачити конфігураційні параметри;
          - відокремити код підготовки даних, навчання та оцінювання.​

4. Автоматизація обчислення метрик моделі Студент повинен автоматично обчислювати метрики відповідно до типу задачі.
          - Для класифікації: Accuracy, Precision, Recall, F1-score, ROC-AUC, Confusion Matrix
          - Для регресії: MAE, MSE, RMSE, R², MAPE
          - Для кластеризації: Silhouette Score, Davies–Bouldin Index, Calinski–Harabasz Index
          - Для виявлення аномалій: Precision@anomaly, Recall@anomaly, F1-score, ROC-AUC або PR-AUC Потрібно пояснити, чому вибрані саме ці метрики.

5. Автоматизація контролю якості даних у конвеєрі Окрім метрик моделі, необхідно включити базові перевірки даних:
          - частка пропусків;
          - кількість дублікатів;
          - коректність типів даних;
          - наявність викидів;
          - частка аномальних або рідкісних значень;
          - баланс класів;
          - відповідність діапазонів значень бізнес-логіці. Результати перевірок повинні або:
          - виводитися у звіт,
          - або зберігатися у файл,
          - або логуватися у консоль / журнал виконання.

6. Збереження результатів оцінювання Необхідно реалізувати автоматичне збереження:
           - значень метрик;
           - параметрів запуску;
           - часу виконання;
           - версії датасету або назви файлу;
           - назви алгоритму;
           - короткого висновку про якість результату. Формат збереження може бути таким: .json, .csv, .txt, таблиця в БД, MLflow / DVC / інший інструмент трекінгу експериментів​

7. Аналіз стабільності результатів Потрібно показати, що оцінювання не є випадковим. Для цього бажано використати:
           - train/test split із фіксацією random_state;
           - cross-validation;
           - repeated validation;
           - порівняння кількох запусків моделі. Студент має показати, що метрики:
           - відтворюються;
           - можуть бути порівняні між моделями;
           - придатні для прийняття рішення щодо вибору алгоритму.​

8. Інтерпретація результатів для бізнес-задачі Потрібно не лише навести метрики, а й пояснити:
           - чи достатня якість моделі для практичного використання;
           - які ризики виникають при низьких значеннях метрик;
           - які метрики є пріоритетними саме для даної бізнес-теми;
           - які покращення можна запропонувати. Наприклад:
           - для кредитного скорингу критичними можуть бути Recall та ROC-AUC;
           - для прогнозу попиту важливими можуть бути MAE та MAPE;
           - для фрод-моніторингу — Recall, Precision та PR-AUC.

  Очікувані результати та критерії оцінювання:
   - (1 бал) Архітектура конвеєру даних – представлено повну схему або опис етапів конвеєру від входу даних до збереження результатів оцінювання


  - (1 бал) Автоматизація обробки даних – реалізовано послідовний запуск модулів підготовки, трансформації та подачі даних у модель


  - (1 бал) Автоматичне обчислення метрик моделі – реалізовано коректне обчислення метрик відповідно до типу задачі машинного навчання


  - (1 бал) Автоматичний контроль якості даних – включено перевірки якості вхідних даних та подано результати контролю


  - (1 бал) Збереження та відтворюваність результатів – результати метрик, параметри запуску та артефакти збережено у придатному форматі


  - (1 бал) Інтерпретація результатів для бізнес-задачі – пояснено практичне значення отриманих метрик та їх вплив на прийняття рішень

      Разом: 6 балів
